## Summary

This phase polishes four features added in Phase 1: the collection grid +/- steppers, tradelist price editor, wantlist/tradelist remove actions, and the deck wizard. All features exist and function — no new server capabilities are required except: (1) joining `card_prices` to the wishlist list query, (2) adding a `card.legends` endpoint to replace the chained pagination hack, and (3) surfacing tier badge data from `decks.tier` field that the riftdecks scraper already populates.

The dominant pattern for this phase is **optimistic UI with undo toasts**. Sonner already exists in the project. The implementation challenge is managing rollback state correctly when multiple rapid taps occur. The secondary challenge is long-press detection in React — no library needed, a `useRef` + `setTimeout` pattern is idiomatic.

The deck wizard improvements are mostly additive UI steps: a preview step before trending deck import, a summary step after legend-path creation, and search/filter controls on the legend picker. The non-English deck name translation requires a client-side decision: use the `Intl` API or a simple heuristic to detect non-Latin scripts and apply a fallback naming strategy without external APIs.

**Primary recommendation:** Implement optimistic updates using tRPC's `utils.collection.list.setData` cache mutation, not just `invalidate()`. This is the correct pattern for instant UI feedback. For long-press, use a 500ms threshold with `touchstart`/`mousedown` + `clearTimeout` on release.

### Pattern 1: Optimistic UI with tRPC Cache Mutation

**What:** Update local cache immediately on user action, trigger server mutation, rollback on error.
**When to use:** Any stepper tap (add/remove copy), where round-trip latency would feel sluggish.

```typescript
// Source: tRPC docs — optimistic updates via setData
const utils = trpc.useUtils();

const addMutation = trpc.collection.add.useMutation({
  onMutate: async ({ cardId }) => {
    // Cancel in-flight refetches
    await utils.collection.list.cancel();
    // Snapshot for rollback
    const prev = utils.collection.list.getData(queryInput);
    // Optimistically bump count in cached pages
    utils.collection.list.setData(queryInput, (old) => {
      if (!old) return old;
      return {
        ...old,
        pages: old.pages.map((page) => ({
          ...page,
          items: page.items.map((item) =>
            item.card.id === cardId
              ? { ...item, _optimistic: true }  // mark for display
              : item,
          ),
        })),
      };
    });
    return { prev };
  },
  onError: (_err, _vars, ctx) => {
    // Roll back
    if (ctx?.prev) utils.collection.list.setData(queryInput, ctx.prev);
    toast.error('Failed to add copy');
  },
  onSettled: () => {
    void utils.collection.list.invalidate();
  },
});
```

**Important note on infinite queries:** `setData` on infinite queries requires working with `pages[]`. The optimistic count shown in the grid is derived from `cardMap` grouping — bump optimistically by inserting a phantom entry into the relevant page's `items` array with a temporary id (`crypto.randomUUID()`).

### Pattern 2: Sonner Undo Toast (5-second window)

**What:** After a destructive action, show a toast with an "Undo" button. If user taps Undo within 5s, cancel the mutation or fire a restore mutation.

```typescript
// Source: sonner docs — toast with action
import { toast } from 'sonner';

// After remove mutation succeeds:
const toastId = toast('Copy removed', {
  duration: 5000,
  action: {
    label: 'Undo',
    onClick: () => {
      // Re-add the copy that was removed
      addMutation.mutate({ cardId, variant: removed.variant, condition: removed.condition });
      toast.dismiss(toastId);
    },
  },
});
```

**Critical detail for undo:** The undo must know WHICH copy was removed (its `variant` and `condition`) to re-add correctly. Capture this before calling the remove mutation. Store `{ id, cardId, variant, condition }` as rollback context.

**Pattern for remove with undo:** Fire the remove mutation AFTER showing the toast (not before). This allows the undo to `toast.dismiss` + skip the mutation entirely, avoiding the round-trip. If undo is NOT clicked, the actual mutation fires when the toast auto-dismisses.

Alternative approach (simpler, recommended): fire the mutation immediately but keep the removed copy's data in state for re-add. Both approaches are valid; the deferred-fire approach is cleaner UX.

### Pattern 6: Non-Latin Deck Name Detection

**What:** Detect and handle Chinese/non-Latin deck names from riftdecks.com.
**Recommended approach (client-side, no external API):**

```typescript
function isNonLatinScript(text: string): boolean {
  // Matches CJK Unified Ideographs block (Chinese, Japanese, Korean)
  return /[\u4E00-\u9FFF\u3400-\u4DBF\uF900-\uFAFF]/.test(text);
}

function getDisplayName(rawName: string, legend?: string, domain?: string): {
  primary: string;
  subtitle: string | null;
} {
  const cleaned = rawName.replace(/^\[RD\]\s*/, '');
  if (isNonLatinScript(cleaned)) {
    const fallback = legend
      ? `${legend.split(' - ')[0]} ${domain ?? 'Deck'}`
      : 'Imported Deck';
    return { primary: fallback, subtitle: cleaned };
  }
  return { primary: cleaned, subtitle: null };
}
```

This runs entirely client-side, no translation API needed, no latency. The subtitle shows the original script below the primary name in the wizard and deck list.

### Pattern 7: Legends Endpoint (replacing chained queries)

**What:** Replace the 6-chained `card.list` pagination calls in `deck-wizard.tsx` with a single `card.legends` tRPC query that returns only Legend-type cards.

**New endpoint in `card.router.ts`:**
```typescript
legends: pub
  .query(() => this.cardService.getLegends()),
```

**Service method in `card.service.ts`:**
```typescript
async getLegends(): Promise<LegendCard[]> {
  return this.db
    .select({ id: cards.id, name: cards.name, cleanName: cards.cleanName, domain: cards.domain, imageSmall: cards.imageSmall })
    .from(cards)
    .where(and(eq(cards.cardType, 'Legend'), eq(cards.isProduct, false)))
    .orderBy(cards.name);
}
```

With ~550 cards total and Legends being a small subset (~20-30), this replaces 6 round-trips with 1.

### Pattern 8: Market Price Join in Wishlist Query

**What:** The `wishlist.service.ts` `list()` method currently does NOT join `card_prices`. To show market price on wantlist/tradelist cards, add a LEFT JOIN.

```typescript
// In wishlist.service.ts list():
.leftJoin(cardPrices, eq(cards.id, cardPrices.cardId))
// Add to select:
card_marketPrice: cardPrices.marketPrice,
card_foilMarketPrice: cardPrices.foilMarketPrice,
```

The `isPublic` field on wishlist entries determines if the entry is normal (non-foil) or foil — but wishlists don't store `variant`. Use `marketPrice` for all wantlist cards (non-foil default) and let the tradelist pre-fill the price input from `marketPrice`.

**For tradelist price pre-fill:** When user opens the price editor, pre-fill `priceInput` with `entry.card.marketPrice ?? ''`.

### Anti-Patterns to Avoid

- **`invalidate()` only for optimistic UI:** `invalidate()` after every tap causes visible flash/rerender. Always use `setData` for the optimistic update + `invalidate` in `onSettled` for eventual consistency.
- **Storing undo state in global context:** Keep undo rollback data in closure scope of the toast callback. No Redux/Zustand needed.
- **Bottom sheet for copy picker:** Overkill for a 2-5 item list. Use an inline popover anchored to the button.
- **`useEffect` for long-press cleanup:** Clean up with `onMouseLeave`/`onTouchCancel` instead.
- **Translation API calls for deck names:** The user base is Honduran (Spanish); Chinese names should fall back to legend+domain, not call an external API.

## Common Pitfalls

### Pitfall 1: Infinite Query `setData` Type Mismatch

**What goes wrong:** `utils.collection.list.setData` for infinite queries expects `{ pages: Page[], pageParams: unknown[] }`, not a flat array. Passing the wrong shape silently fails and the optimistic update doesn't appear.

**How to avoid:** Always spread `old` and map over `old.pages`. Use TypeScript strict mode — the type error will surface immediately.

**Warning signs:** Optimistic update not visible even though `setData` was called without errors.

### Pitfall 2: Long-Press Fires Both Tap and Long-Press Handlers

**What goes wrong:** On mobile, `touchend` fires after `touchstart`, which triggers the tap handler even when the user long-pressed (opening the picker).

**How to avoid:** Track whether the long-press timer fired using a ref flag. Suppress the normal tap handler if `longPressTriggered.current === true`.

```typescript
const longPressTriggered = useRef(false);

const handlePressStart = () => {
  longPressTriggered.current = false;
  longPressTimer.current = setTimeout(() => {
    longPressTriggered.current = true;
    openVariantPicker();
  }, 500);
};

const handleTap = () => {
  if (longPressTriggered.current) return; // swallow tap
  handleAdd();
};
```

### Pitfall 3: Undo Re-adds a Copy with Wrong State

**What goes wrong:** After removing copy A (NM, normal) and tapping Undo, the re-add mutates with the default `{ variant: 'normal', condition: 'near_mint' }` — but the removed copy might have been LP or alt_art.

**How to avoid:** Capture `{ variant, condition }` from the removed copy before the mutation and pass them to the undo's `addMutation.mutate()` call.

### Pitfall 4: Sonner `action` Button Not Dismissed After Click

**What goes wrong:** Clicking "Undo" closes the toast but the action dismisses by default only in some sonner versions. Explicitly call `toast.dismiss(toastId)` in the action `onClick`.

**How to avoid:** Always capture the toast ID returned by `toast(...)` and call `toast.dismiss(id)` inside the undo handler.

### Pitfall 5: Wishlist `list()` Returns Null `marketPrice` for Cards Without Prices

**What goes wrong:** Not all cards have `card_prices` rows. The LEFT JOIN returns `null` for `marketPrice`. Displaying `$null` or `$undefined` breaks the UI.

**How to avoid:** Always null-coalesce: `entry.card.marketPrice ?? null`. Show the market price badge only when the value is non-null.

### Pitfall 6: Tier Badge Field May Not Be in Current Deck Schema

**What goes wrong:** The context says riftdecks scraper reads tier badges, but the `decks` table may not have a `tier` column exposed through the `browse` query.

**How to avoid:** Check `packages/db/src/schema/decks.ts` and `deck.service.ts`'s `browse()` method before assuming tier data is already surfaced. If the column exists but isn't selected, add it to the query. If it doesn't exist, a migration is needed.

## Code Examples

### Optimistic Count Display in Grid

```typescript
// Source: derived from tRPC infinite query cache pattern
// Track optimistic additions: cardId -> delta (positive = added, negative = removed)
const [optimisticDeltas, setOptimisticDeltas] = useState<Map<string, number>>(new Map());

// In render:
const displayCount = copies.length + (optimisticDeltas.get(card.id) ?? 0);

// In handleAdd (before mutate):
setOptimisticDeltas((prev) => {
  const next = new Map(prev);
  next.set(cardId, (next.get(cardId) ?? 0) + 1);
  return next;
});

// In onSettled:
setOptimisticDeltas((prev) => {
  const next = new Map(prev);
  next.delete(cardId);
  return next;
});
// Then: void utils.collection.list.invalidate();
```

This approach is simpler than mutating the infinite query pages directly and avoids the type complexity.

## Open Questions

1. **Does the `decks` table have a `tier` column?**
   - What we know: The riftdecks scraper reads tier badge CSS classes (S/A/B/C). Phase 1 decision log says "Riftdecks.com scraper reads tier badge CSS classes."
   - What's unclear: Whether `tier` is stored in the DB schema and returned by `deck.browse`.
   - Recommendation: Read `packages/db/src/schema/decks.ts` and `deck.service.ts` browse() during planning. If tier column exists but not selected, add it to browse query. If missing entirely, add a new nullable column + migration in Wave 1.

2. **Does collection.list need a card_prices join for the "Total: $X" badge?**
   - What we know: The `list()` query in `collection.service.ts` does NOT currently join `card_prices`.
   - What's unclear: Whether adding the join is acceptable performance-wise (it adds a LEFT JOIN to every list call).
   - Recommendation: Add the LEFT JOIN. The `card_prices` table has an index on `card_id`. The join cost is low. Alternatively, compute the total from the existing stats endpoint (but it's not filter-aware).

3. **For wantlist market price: which price field to use?**
   - What we know: Wishlist entries don't store `variant`. Cards can be foil or non-foil.
   - What's unclear: Whether to always show `marketPrice` (non-foil) or check something else.
   - Recommendation: Always use `marketPrice` (non-foil) for wantlist. If user wants a foil copy, the price shown is slightly wrong, but wishlists don't have a variant field and adding one is out of scope for this polish phase.
