## Summary

Phase 01.2 upgrades the existing two-panel deck editor into a zone-aware, collection-integrated, analytics-rich, and suggestion-powered smart builder. The core editor (`deck-card-editor.tsx`) needs zone tabs (Main/Runes/Champion/Sideboard), the service layer needs a suggestion endpoint plus per-zone validation, and the deck list needs buildability percentage and draft/complete status.

The critical game-rules finding: Riftbound Constructed uses **40-card main, 12 runes, 1 legend, 3 copies max per card, and 8-card sideboard** (or 0 — no in-between). This replaces the current permissive schema (MAX 60 cards, 12 copies per card) which was wrong for the Riftbound format. The DB schema for `deck_cards` needs a `zone` column and the `decks` table needs a `status` column (draft/complete).

The suggestion engine is the highest-complexity new piece. It lives entirely server-side as a new tRPC procedure, uses a hybrid scoring approach (domain + curve + co-occurrence + keywords), and caches results in Redis. The keyword tagging cron follows the exact `@nestjs/schedule` pattern already used in `deck-sync.service.ts`.

**Primary recommendation:** Implement in four sequential work streams: (1) DB migration + zone-aware schema, (2) format validation overhaul in service + Zod schemas, (3) suggestion engine endpoint with caching, (4) editor/UI refactor with analytics.

## Riftbound Game Rules (VERIFIED)

This section documents findings from official Riftbound sources — these constrain ALL validation logic.

### Official Deck Construction Rules

| Zone | Size | Notes |
|------|------|-------|
| Main Deck | Exactly 40 cards | No minimum deviation |
| Rune Deck | Exactly 12 runes | One per domain slot (6+6 recommended) |
| Legend | Exactly 1 | Always in play, not drawn |
| Sideboard | 0 or exactly 8 | No in-between values are legal |
| Total (Main+Runes+Legend) | 53 cards | Sideboard separate |

### Copy Limits

- **Maximum 3 copies** of any named card across main deck + sideboard combined
- This applies to ALL card types including Runes
- A Legend card counts toward the 3-copy limit if the same named card appears in deck as a unit
- Signature cards (Signature Unit, Signature Spell, Signature Gear) — official limit is **1 copy** (unique to each Legend)

**Critical finding:** The current `MAX_COPIES_PER_CARD = 12` and `MAX_DECK_CARDS = 60` in `deck.service.ts` are WRONG for Riftbound format. These must be replaced with zone-aware validation.

### Sideboard Rule Detail (HIGH confidence — from official search results)
- Sideboard must be **exactly 0 or exactly 8 cards**
- Sideboard cards count toward the 3-copy limit combined with main deck
- Sideboard must follow all standard deckbuilding constraints (domain, copy limits)

**Source:** Official search result from `riftbound.leagueoflegends.com` tournament rules, January 2026 update.

### Pattern 2: Format Validation Logic (Server-Side)

```typescript
// In deck.service.ts — replace validateCardEntries()
interface DeckZones {
  main: Map<string, number>;      // cardId -> quantity
  rune: Map<string, number>;
  champion: Map<string, number>;
  sideboard: Map<string, number>;
}

function validateFormat(zones: DeckZones): string[] {
  const errors: string[] = [];

  const mainTotal = [...zones.main.values()].reduce((s, q) => s + q, 0);
  const runeTotal = [...zones.rune.values()].reduce((s, q) => s + q, 0);
  const championTotal = [...zones.champion.values()].reduce((s, q) => s + q, 0);
  const sideboardTotal = [...zones.sideboard.values()].reduce((s, q) => s + q, 0);

  if (mainTotal !== 40) errors.push(`Main: ${mainTotal}/40`);
  if (runeTotal !== 12) errors.push(`Runes: ${runeTotal}/12`);
  if (championTotal !== 1) errors.push('Need 1 champion');
  if (sideboardTotal !== 0 && sideboardTotal !== 8)
    errors.push(`Sideboard: ${sideboardTotal} (must be 0 or 8)`);

  // 3-copy limit across main + sideboard combined
  const allCardIds = new Set([
    ...zones.main.keys(), ...zones.rune.keys(),
    ...zones.champion.keys(), ...zones.sideboard.keys(),
  ]);
  for (const cardId of allCardIds) {
    const totalCopies = (zones.main.get(cardId) ?? 0)
      + (zones.sideboard.get(cardId) ?? 0);
    if (totalCopies > 3) errors.push(`Too many copies: ${cardId}`);
    // Signature cards: max 1 copy (enforce on type lookup)
  }

  return errors;
}
```

### Anti-Patterns to Avoid

- **Modifying the card.list query for ownership:** adds join overhead on every search. Use a separate bulk ownership query keyed by card IDs in view.
- **Re-validating format on every keystroke server-side:** format validation runs client-side in real-time; the server validates only on save (setCards).
- **Storing analytics in DB:** all analytics derive from card attributes already in deck. Compute client-side.
- **Suggestions on every deck change:** explicitly decided to use manual refresh button. Do NOT make suggestions reactive to deck changes.
- **Monolithic setCards for all zones:** the new `setCards` input must include zone per entry — the existing schema sends no zone, causing all cards to land in main.

## Common Pitfalls

### Pitfall 1: Zone Constraint on uniqueIndex
**What goes wrong:** The current `uniqueIndex('idx_deck_cards_deck_card').on(table.deckId, table.cardId)` prevents the same card from being in both the main deck and the sideboard. This must change to `(deckId, cardId, zone)` — but that then allows the same card 3+ times in different zones, requiring service-level copy limit enforcement across zones.
**How to avoid:** The migration adds `zone` column and drops/recreates the unique index. Copy-limit validation in `validateZones()` aggregates across main + sideboard (rune zone counts separately since copies of rune cards by domain, not by name).
**Warning signs:** FK violation or duplicate key errors when trying to add a card that's already in another zone.

### Pitfall 2: Signature Card Copy Limit
**What goes wrong:** Signature Unit/Spell/Gear cards are unique to each Legend — only 1 copy allowed. Without a check, users could add 3 copies of a Signature card.
**How to avoid:** In `validateZones()`, check `cardType` for signature types and enforce max 1 copy. The card's `cardType` is available on the loaded deck card rows.
**Warning signs:** A user adds 3x "Jinx's Fishbones" (Signature Gear) and it validates as legal.

### Pitfall 3: Sideboard "0 or 8" Constraint
**What goes wrong:** Riftbound requires sideboard to be exactly 0 or exactly 8 cards — not 1-7. Saving a deck with 5 sideboard cards should mark it "Draft" and show "Sideboard: 5 (must be 0 or 8)".
**How to avoid:** Add the 0-or-8 check explicitly in `validateFormat()`. A deck can be saved with any sideboard size — just marked Draft.
**Warning signs:** Users can create a "valid" deck with 3 sideboard cards.

### Pitfall 4: energyCost is String in Card Data
**What goes wrong:** The `riftbound-tcg-data` JSON has `energyCost: "6"` as a string. The DB schema stores it as `integer`. The analytics computation must parse to int; cards with `energyCost: null` (Legend, some products) must be bucketed separately or excluded from curve.
**How to avoid:** `const cost = parseInt(entry.card.energyCost ?? '0', 10)` with null guard. Legends and runes typically have cost 0.
**Warning signs:** Energy curve chart shows NaN bars or crashes on null energyCost.

### Pitfall 5: tRPC cache invalidation after setCards with zone
**What goes wrong:** After saving, `utils.deck.getById.invalidate({ id: deckId })` must be called. If analytics depend on a separate query that's not invalidated, stale data shows in analytics.
**How to avoid:** All analytics derive from `deck.getById` — single invalidation point covers everything. No additional queries for analytics means no additional cache to manage.

### Pitfall 6: Wizard Auto-Fill with Wrong Copy Limits
**What goes wrong:** `buildStarterCardsFromPool()` in `deck-wizard.tsx` currently assigns `qty = 2` for Common/Uncommon cards. Riftbound max is 3, not 2 or 12. The wizard must respect zone structure (main: 40, rune: 12, champion: 1).
**How to avoid:** Rewrite wizard auto-fill to build zone-aware card lists. Main pool fills to 39 (excluding champion), champion zone adds the legend, rune zone adds 12 matching runes.

### Buildability Percentage Query Pattern

```typescript
// In deck.service.ts — new method
async getBuildability(userId: string, deckId: string): Promise<{
  owned: number; total: number; pct: number; missingCardIds: string[];
}> {
  // Get deck cards (all zones)
  const deckCardRows = await this.db.select(...)
    .from(deckCards).where(eq(deckCards.deckId, deckId));

  // Get user owned card IDs
  const owned = await this.db
    .select({ cardId: collections.cardId })
    .from(collections)
    .where(eq(collections.userId, userId));

  const ownedSet = new Set(owned.map(o => o.cardId));
  const total = deckCardRows.length;
  const ownedCount = deckCardRows.filter(dc => ownedSet.has(dc.cardId)).length;

  return {
    owned: ownedCount,
    total,
    pct: total > 0 ? Math.round((ownedCount / total) * 100) : 0,
    missingCardIds: deckCardRows
      .filter(dc => !ownedSet.has(dc.cardId))
      .map(dc => dc.cardId),
  };
}
```

## Open Questions

1. **Signature card 1-copy enforcement**
   - What we know: Signature Unit/Spell/Gear are unique to each Legend — by game design, only 1 exists per Legend, so having 3 copies is physically possible but unusual
   - What's unclear: Whether official rules state max 1 Signature card or rely on scarcity (there's only 1 Signature card per Legend in the set)
   - Recommendation: Enforce max 1 per Signature type (cardType includes 'Signature') in validation — safe and conservative

2. **Rune copy limit: by card name or by domain?**
   - What we know: Runes are domain-locked (Fury Rune, Calm Rune etc.) with 12 total in rune zone
   - What's unclear: Whether you can run 12 Fury Runes or whether it's 6+6 enforced
   - Recommendation: Enforce the 3-copy-per-named-card rule on runes too (so max 3 Fury Runes), let users fill remaining slots freely. The 12-rune total is the hard constraint.

3. **`card_keywords` storage approach**
   - What we know: Keywords must be tagged periodically via cron
   - What's unclear: Whether to add a `keywords` JSON column to `cards` table or create a separate `card_keywords` junction table
   - Recommendation: Add `keywords` text array column to `cards` table — simpler, avoids join in suggestion query. Migrate in Wave 0.

4. **Suggestion caching TTL**
   - What we know: Redis caching needed; suggestions refresh only on manual button press
   - Recommendation: 10 minutes TTL keyed by `deck:suggest:{deckId}:{mode}`. Cache is invalidated on deck save (setCards).
